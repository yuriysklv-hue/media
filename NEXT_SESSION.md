Сделать репо `yuriysklv-hue/media` закрытым, не сломав автодеплой. **Порядок важен — сначала SSH, потом приватность:**
1. В панели Timeweb (приложение → Настройки → подключение репозитория) переключить способ доступа на **SSH**: Timeweb выдаёт/просит публичный SSH-ключ.
2. Добавить этот ключ в GitHub: репо → Settings → **Deploy keys** → Add deploy key (доступ read-only достаточен), URL репо в Timeweb — `git@github.com:yuriysklv-hue/media.git`.
3. Прогнать тестовый деплой (кнопка ▶) — убедиться, что клонирование по SSH работает **до** смены видимости.
4. GitHub → Settings → General → Danger Zone → **Change visibility → Private**.
5. Снова прогнать деплой и проверить автодеплой пушем в `main`.
Заметки: рабочие сессии Claude ходят в репо через GitHub App — приватность им не мешает; вебхук автодеплоя тоже сохраняется. Если у Timeweb «SSH-способ» выглядит иначе (например, только через их GitHub-интеграцию) — задокументировать фактический путь в CLAUDE.md.

Проверено в контенте: это обычный `<p>`, без markdown-blockquote (`>`), поэтому ни `<blockquote>`, ни `<q>`, ни schema не появляются. `blockquote` в стилях есть только для карточек мнений (`.op-card blockquote`), к телу статьи не относится.
  - **✅ Спроектировано (16.07.2026) — развилки закрыты, ждём реализации.** Разбор корпуса (40 боевых новостей): кавычки «» **перегружены** — это и прямая речь («Мы хотели убедиться…»), и термины/скейр-кавычки («CTV», «щупальца», «демонтировали»); речевая атрибуция — в ~25/40 файлов. Атрибуция стоит в 3 позициях (после цитаты `«…», — подчеркнул X`; **внутри** — расщеплённая `«…текст — пояснила X. — ещё текст»`; перед `X добавил: «…»`) + пересказ `по словам X` (не прямая речь). Вывод: **надёжная разметка возможна только там, где известен спикер — в момент написания (агент); автораспознавание на сайте регуляркой по «» ненадёжно** (не отличит речь от термина, не режет расщеплённые цитаты).

  **Зафиксированные решения:**
  1. **Стадия — агент размечает, сайт рендерит.** `media-agents` (Writer/Enricher) отдаёт разметку в теле; `media-site` только стилизует. Без хрупкого парсинга.
  2. **Формат — вариант B: инлайновый `<q cite="URL">` сырым HTML в `.md`** (НЕ `.mdx`, НЕ кастомный компонент). Причина: главный SEO-скачок — из «пунктуации» в семантическую цитату — даёт уже `<q>`; a `.md` сохраняет свойство «контент не может сломать сборку» (у `.mdx`/компонента любой кривой символ роняет весь `astro build` → нужен был бы MDX-компиляционный гейт в агенте). Отклонены: A (MDX-компонент `<Quote speaker… role… source…>` + schema.org `Quotation`/`spokenByCharacter`) — даёт структурного спикера, но rich-поддержка сегодня слабая → оставлен как **апгрейд-путь на вырост**; поле `quotes[]` во фронт-маттере — дублирует прозу.
  3. **Что размечаем — только прямую речь спикеров.** НЕ термины/скейр-кавычки, НЕ пересказ `по словам X` без прямой цитаты.
  4. **Внешний вид — без изменений** (канон «минимал»): визуально A и B неразличимы, ценность разметки машинная, а не визуальная. Выносные **pull-quotes — вне объёма**, отдельной задачей позже при желании.
  5. **schema.org в MVP не добавляем** — у B спикер остаётся прозой, structured-`Quotation` без него малоценен при слабой rich-поддержке. Апгрейд до schema (= путь A) — потом, если понадобится.

  **Конвенция разметки (контракт media-agents → media-site):**
  - Типографские «» **оставляем как есть** (агент уже их пишет; в 40 статьях они есть) — оборачиваем содержимое в `<q cite="<url источника>">…</q>`, `cite` = URL первоисточника (обычно `source.url`).
  - Пример (после): `«<q cite="https://adexchanger.com/…">Мы хотели убедиться, что в первую очередь думаем о потребительском опыте</q>», — пояснила Финштейн.`
  - **Расщеплённая цитата → по одному `<q>` на каждый речевой фрагмент**, атрибуция между ними остаётся снаружи: `«<q cite="…">У нас есть привычка упускать из виду…</q> — пояснила Финштейн. — <q cite="…">Мы хотели убедиться…</q>»`.
  - Сайт гасит авто-кавычки браузера у `<q>` (`.article-body q::before,::after { content: none }`), иначе двойные «„…"». Видимый текст = ровно как напечатан.

  **Разбивка работ:**
  - **media (сайт), маленькая:** ~3–5 строк CSS в `global.css` (`.article-body q`: нейтрализовать авто-кавычки, при желании — лёгкий акцент; по канону визуально не выделяем) + проверить, что сырой `<q>` проходит через markdown (как уже проходит `<small>`). Сам по себе бесполезен, пока в контенте нет `<q>`, поэтому мержить синхронно с первым агентским выводом.
  - **media-agents, основная:** Writer/Enricher эмитит `<q cite>` по правилам выше; обновить промпты + корпус good/bad примерами; QA — проверка, что размечена речь, а не термины. Спека готова — переносить в сессию `media-agents` (add_repo).

### Врезка: как SEO-задачи связаны с выдачей в LLM-чатах

Чтобы попадать в ответы ChatGPT, Perplexity, Gemini, YandexGPT и т.п., работает **не отдельная «магия для ИИ», а обычная индексация + структура под цитирование.** Два канала:

1. **Обучающий корпус** (боты `GPTBot`/`ClaudeBot`/`CCBot` из `robots.txt`) — модель обучается на скрейпе с лагом в месяцы; для нового сайта в моменте почти не работает. У нас пущены «на вырост».
2. **Живой веб-поиск в момент ответа (RAG)** — главный канал для нового сайта: чат ищет по вебу, читает верхние страницы, цитирует со ссылкой. Почти все грундятся на существующих поисковых индексах:
   - ChatGPT Search, Perplexity, Copilot → **Bing** (+ свои краулеры);
   - Gemini / AI Overviews → **Google**;
   - YandexGPT / Нейро → **Яндекс**.

**Практический вывод:** индексация в Google + **Bing** + Яндексе = фундамент LLM-видимости (отсюда обязательный Bing Webmaster Tools в задаче A). `llms.txt`/`llms-full.txt` — молодой стандарт, поддержка у движков слабая, держим «на вырост». Чистая структура (H1→H2, первый абзац = саммари, факты с источниками, размеченные цитаты) → LLM легче вытащить готовую цитату именно из нас.

- **Ответ на вопрос владельца.** Дайджест — это **полноценная статья** в коллекции `digest` (своя страница `/article/<week>/`, напр. `/article/2026-w27/`, поля `highlights[]` и `week` во фронт-маттере). Блок «Дайджест недели» в правом сайдбаре главной — **тизер**: ссылка на статью-дайджест (`.d-head`) + буллеты из `highlights`.
- **Синхронизация с пайплайном — через контент-коллекцию, автоматическая.** Агент `media-agents` кладёт `.md` в `src/content/digest/`; при мерже PR главная сама берёт самый свежий (`all.find(e => e.collection==='digest')`). Сегодняшний дайджест после мержа появится: (1) тизер на главной, (2) лента `/digest`, (3) своя статья. На главной показывается только один (свежий) дайджест — норма.
- [ ] **Мелкая проверка (после первого боевого дайджеста):** убедиться, что тизер и буллеты `highlights` корректно отрисовались, ссылка ведёт на статью, дата/`week` верные. Разовая проверка, не код-задача.

### 8.4. Логика «Главное сейчас» vs «Свежее» — как устроено (справка, действий не требуется)
- **Ответ на вопрос владельца** (`index.astro`): **Hero** — свежий `featured` (иначе свежий не-новостной). **«Главное сейчас»** = 3 самые свежие **новости** кроме hero (live-точка, если материал младше 3 ч). **«Свежее»** = следующие материалы (**news + reviews**) кроме hero и тех трёх, до 6 штук. Пересечений нет: «Главное» — верхушка новостной ленты, «Свежее» — продолжение. Работает by design; менять только если владелец захочет иную витрину (тогда завести отдельную задачу).
